# @plughub/sdk — Integration SDK

## What this package is

The SDK that any external agent uses to participate in the PlugHub Platform pool.
Implements the 9 responsibilities from spec section 4.6a.

## What this package is NOT

- Not an agent framework — does not define how the agent reasons
- Not an MCP Server — does not expose tools to the BPM
- Not business logic — only the execution contract

## Main files

- `adapter.ts` — PlugHubAdapter + drivers (GenericMCPDriver, BedrockDriver, etc.)
- `agent.ts`   — definePlugHubAgent (wrapper function) + automatic lifecycle
- `lifecycle.ts` — agent_login, agent_ready, agent_busy, agent_done via MCP
- `certify.ts` — certification logic (spec 4.6e)
- `portability.ts` — verify-portability logic (spec 4.6h)
- `observability.ts` — propagation of plughub.session_id as trace root
- `proxy/` — proxy sidecar for external agents (spec 4.6k)
  - `server.ts` — HTTP proxy that intercepts MCP calls, validates JWT permissions, writes async audit
  - `config.ts` — loads and validates proxy_config.yaml
  - `circuit-breaker.ts` — timeout + error_clear fallback mode
- `cli/` — CLI tool entry points (certify, regenerate, verify-portability, proxy, skill-extract)

## Dependencies

- `@plughub/schemas` — source of truth for contracts
- `@modelcontextprotocol/sdk` — MCP communication with mcp-server-plughub
- `zod` — schema validation
- `commander` — CLI

## Proxy sidecar — architecture decision (spec 4.6k, strategy section 11)

External agents (LangGraph, CrewAI, etc.) that do not use the SDK call domain MCP Servers
directly, bypassing permission validation and audit logging. The proxy sidecar solves this:

| Agent type | Interception | Extra process |
|---|---|---|
| Native (uses SDK) | In-process via PlugHubAdapter | None |
| External (LangGraph, CrewAI) | Sidecar on localhost:7422 | `plughub-sdk proxy` |
| GitAgent (from regenerate) | In-process via SDK (generated code) | None |

**How the sidecar works per MCP call** (overhead < 1ms total):
1. Validates tool against `permissions[]` embedded in session_token JWT — local, ~0.1ms, zero network
2. Writes audit event to local buffer — async, ~0ms (background thread drains to Kafka)
3. Forwards to real domain MCP Server endpoint
4. Returns response to agent

**proxy_config.yaml** — generated by `plughub-sdk regenerate` alongside the agent output:
```yaml
port: 7422
session_token_env: PLUGHUB_SESSION_TOKEN
audit_buffer_size: 1000
audit_flush_interval_ms: 500
circuit_breaker:
  timeout_ms: 50
  mode_on_failure: error_clear   # returns clear error to agent; never silently passes
routes:                           # derived from agent-type permissions[]
  mcp-server-crm:   ${MCP_CRM_URL}
  mcp-server-telco: ${MCP_TELCO_URL}
```

Start with: `plughub-sdk proxy --config ./output/proxy_config.yaml`

## Invariants for this package

- PlugHubAdapter is the only adaptation interface — never create inline adaptation
- Drivers are separate from the adapter — adapter is environment-agnostic
- The lifecycle (login/ready/busy/done) is managed automatically by the SDK
- Never expose session_id or tenant_id directly to agent code
- context_map and result_map are bidirectional — the same object serves input and output
- External agents MUST route all domain MCP calls through the proxy sidecar — never directly
- Proxy permission validation uses the JWT local signature only — zero network calls per MCP call
- Audit writes are always async — proxy never blocks an MCP call waiting for audit ack

## What never to do

- Never access context_package directly in agent code — always via adapter
- Never call agent_done manually — the wrapper does it automatically
- Never create an inline driver — use package drivers or create under drivers/
- Never import from @plughub/schemas in agent code — only in the SDK
- Never deploy an external agent without the proxy sidecar running — calls bypass audit and permissions
- Never make the proxy do synchronous audit writes — audit must be async (buffer + background thread)
- Never validate permissions via network call per MCP call — embed permissions in JWT, validate locally
