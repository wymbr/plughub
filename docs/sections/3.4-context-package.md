# 3.4 Context Package + 3.4a Conversation Insights

> Fonte: PlugHub spec v24.0

3.4 Context Package — Sessão como Fonte de Verdade

O Context Package não é um objeto construído no momento do handoff — é o
próprio estado da sessão acumulado turno a turno no Redis. Qualquer
agente que assumir a conversa lê a sessão e começa com o contexto
disponível. Não há componente responsável por “entregar” o contexto —
ele está sempre acessível.

Campos acumulados na sessão durante o atendimento:

{ conversation_summary, intent_history, sentiment_trajectory,

customer_profile: { tier, ltv, churn_risk },

attempted_resolutions, handoff_reason,

conversation_insights: [ ...insights ativos do cliente ],

pending_deliveries: [ ...pendências ativas do cliente ] }

O Escalation Engine, ao executar uma mudança de alocação, atualiza
atomicamente o estado da sessão (novo agente, motivo, timestamp),
publica evento em conversations.events e envia mensagem de transição ao
cliente em conversations.outbound. O agente destino lê a sessão no
início do primeiro turno.

Para voz: Warm Transfer com Whisper — agente humano ouve briefing em
áudio. Cliente permanece na linha com hold de no máximo 10 segundos.

3.4a Conversation Insights e Pending Delivery — Modelo Unificado

Conversation Insights e Pending Deliveries são instâncias do mesmo
modelo — diferenciados apenas pela categoria. Ambos têm validade,
prioridade, status e origem. A categoria usa notação hierárquica com
separador ponto (a.b.c). O comportamento de cada categoria é
inteiramente definido por configuração operacional por tenant — a
plataforma não conhece categorias nativamente.

{ item_id, customer_id, tenant_id, category, // e.g.
"outbound.retencao.oferta" or "insight.risco.churn_signal"

content, source, source_session_id, expires_at, priority,

status: pending | offered | accepted | delivered | consumed | expired
| replaced }

Configuração operacional de categoria (por tenant, suporta wildcard):

{ consome_no_uso, notifica_agente, alerta, validade_padrao,
prioridade_padrao, canais_adequados, requer_confirmacao, substituivel }

No início de cada contato, o Routing Engine consulta itens ativos do
cliente e os inclui na sessão. Leitores filtram pelo prefixo de
categoria que precisam: agente filtra insight.*, Notification Agent
filtra outbound.*, Agent Assist filtra insight.historico.* e
insight.conversa.*.

Duas categorias de insight com naturezas distintas

  --------------------- ------------------------ ------------------------
  Dimensão              insight.historico.*      insight.conversa.*

  Origem                Sessões anteriores, CRM, Conversa atual em
                        ERP, sistemas de backend andamento

  Quando gerado         Antes do contato ou em   Durante a sessão atual
                        sessões passadas         via insight_register

  Vida útil             Dias ou semanas —        Duração da conversa —
                        validade configurável    expira no encerramento
                        por pool                 

  Quem gera             Qualquer sistema — BPM,  Agente atual — IA ou
                        CRM, agente anterior     humano via tool MCP

  Carregamento          No início do contato     Tempo real — a cada
                        pelo Routing Engine      chamada de
                                                 insight_register

  Exemplo               Cliente tentou cancelar  Cliente mencionou falha
                        em março — revertido com técnica reportada há 5
                        desconto de 30%          dias, não resolvida
  --------------------- ------------------------ ------------------------

insight.historico.* é memória objetiva de longo prazo do cliente.
insight.conversa.* é memória objetiva da conversa em andamento. Ambos
são fatos — não inferências subjetivas como sentiment ou churn risk, que
pertencem à camada de parâmetros observáveis do AI Gateway. O Agent
Assist apresenta os dois no painel de contexto do cliente, com
procedência e grau de certeza visíveis.

3.4c Caminho de Escrita e Promoção de insight.historico.*

A persistência de insight.historico.* no PostgreSQL ocorre no
encerramento do contato — não no encerramento de cada sessão de agente.
Um contato pode atravessar múltiplas sessões de agente (escalações,
transferências) antes de ser encerrado, e nenhuma perda de dados é
tolerável entre essas transições. Por isso, a escrita direta no
PostgreSQL pelo sistema gerador não é suficiente: uma falha entre
sessões perderia insights registrados na sessão anterior.

O caminho de escrita adota o padrão Kafka-first da plataforma:

  insight_register (MCP tool)
    → 1. grava insight.conversa.* no Redis da sessão
         (imediato — disponível ao Agent Assist no próximo turno)
    → 2. publica evento insight.registered em conversations.events
         (durável — comprometido no log antes do retorno ao agente)

  contact_closed event em conversations.events
    → consumer de persistência filtra todos os insight.registered
      do contato (por session_id e contact_id)
    → promove insight.conversa.* → insight.historico.*
    → persiste no PostgreSQL com TTL configurado por pool (via
      insight_categories do supervisor_config)

Garantias desta decisão:

  - Durabilidade: o evento está no Kafka antes do retorno do MCP call.
    Falha entre sessões de agente não perde nenhum insight.
  - Ciclo de vida correto: a fronteira de persistência é o contato,
    não a sessão de agente. O consumer tem a visão completa do contato.
  - Consistência com o modelo de auditoria: mesmo padrão de
    conversations.events → consumer → PostgreSQL/ClickHouse já usado
    para auditoria e analytics. Nenhum caminho de escrita fora do
    modelo de eventos da plataforma.
  - Idempotência: o consumer usa contact_id + insight_id como chave
    de deduplicação — reprocessamento seguro em caso de falha do
    consumer.

O Routing Engine carrega insight.historico.* do PostgreSQL no início
de cada novo contato do cliente, filtrando por insight_categories
declaradas no pool e pela janela history_window_days.

3.4b Política de Versionamento do Schema do context_package

O schema do context_package evolui ao longo do tempo. Sem política
explícita, mudanças criam três problemas simultâneos: conversas ativas
no Redis têm o schema antigo; agentes novos e antigos coexistem durante
deploy canário; dados históricos no ClickHouse precisam ser consultáveis
junto com dados novos.

  --------------------- ---------------- ---------------------------------
  Tipo de mudança       Compatível sem   Procedimento
                        migração?        

  Adicionar campo com   Sim              Agentes antigos ignoram; agentes
  default                                novos usam default se ausente

  Renomear campo        Nao — breaking   Manter ambos os nomes por minimo
  existente             change           2 sprints (campo antigo
                                         deprecated); remover apenas com
                                         100% dos agentes na versao nova

  Remover campo         Nao — breaking   Mesmo processo de renomear:
                        change           deprecated por N deploys

  Mudar tipo de campo   Nao — breaking   Nunca fazer. Criar campo novo com
                        change           tipo correto e deprecar o antigo

  Adicionar valor a     Com cuidado      Agentes antigos devem ter
  enum                                   fallback para valores
                                         desconhecidos de enum
  --------------------- ---------------- ---------------------------------

O campo schema_version (integer) deve ser adicionado ao context_package.
Cada mudanca de schema incrementa o valor. O Agent Registry suporta
migration function por versao: ao encontrar schema_version antigo, o
agente aplica a migracao antes de processar. O pipeline de deploy inclui
verificacao automatica de compatibilidade de schema antes do canario.
